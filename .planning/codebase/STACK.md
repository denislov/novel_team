# Technology Stack

**Analysis Date:** 2026-04-07

## Languages

**Primary:**
- JavaScript (CommonJS) - installer and runtime-surface generation in `bin/install.js`
- Markdown - command, workflow, agent, skill, and template source bundle under `plugins/novel/`

**Secondary:**
- Python 3.10+ - workflow helper CLIs in `plugins/novel/scripts/*.py`
- JSON - plugin manifests and installer metadata in `package.json`, `.agents/plugins/marketplace.json`, `.claude-plugin/marketplace.json`, and `plugins/novel/.{claude-plugin,codex-plugin}/plugin.json`

## Runtime

**Environment:**
- Node.js `>=20.0.0` - required by `package.json` and used by `bin/install.js`
- Python 3.10+ - effectively required by `plugins/novel/scripts/map_base.py`, `plugins/novel/scripts/novel_state.py`, and `plugins/novel/scripts/chapter_ops.py` because they use modern typing syntax such as `str | None` and `list[str]`
- No browser runtime and no long-lived server process

**Package Manager:**
- npm - implied by `package.json` and the documented `npm test` flow
- Lockfile: none present in the repo root

## Frameworks

**Core:**
- None. The executable code is a vanilla Node CLI plus standalone Python CLIs.

**Testing:**
- Node built-in test runner via `node --test` from `package.json`
- Python `unittest` for script-level tests in `plugins/novel/scripts/test_*.py`

**Build/Dev:**
- No bundler, transpiler, or compile step is configured
- Installation and update are performed by direct file copy and rewrite logic in `bin/install.js`

## Key Dependencies

**Critical:**
- Node built-ins `fs`, `path`, and `os` - core installer I/O, path resolution, and target-directory selection in `bin/install.js`
- Node built-in test modules `node:test` and `node:assert` - installer and asset-contract verification in `tests/*.test.cjs`
- Python stdlib `argparse`, `pathlib`, `json`, `re`, `subprocess`, and `tempfile` - CLI parsing, filesystem scanning, and state/report generation in `plugins/novel/scripts/*.py`

**Infrastructure:**
- `package.json` - source of truth for package metadata, Node engine floor, `bin` entrypoint, and test command
- `plugins/novel/` - the editable source bundle copied into runtime-specific config trees for Claude and Codex

## Configuration

**Environment:**
- No `.env` files or secret-bearing config are used in this repo
- Installer target roots are influenced by `CODEX_HOME` and `CLAUDE_CONFIG_DIR` in `bin/install.js`
- CLI flags `--claude`, `--codex`, `--all`, `--global`, `--local`, and `--config-dir` control install/update/uninstall behavior

**Build:**
- No build config files such as `tsconfig.json`, `vite.config.*`, or `webpack.config.*` are present
- Runtime surfaces are generated by rewriting Markdown/TOML/YAML references in `bin/install.js`

## Platform Requirements

**Development:**
- Any platform with Node.js 20+ and Python 3.10+
- Writable filesystem access for temp directories during tests and for runtime config trees during install
- No database, Docker, or background services required

**Production:**
- Distributed as a local CLI package exposed via `bin/install.js` and `package.json`
- Intended deployment target is the user's Claude or Codex config directory, not a hosted service
- Runtime-generated assets expect support bundles under `~/.claude/novel` or `~/.codex/novel` unless overridden

## Stack Notes

- The repo intentionally avoids third-party npm and Python package dependencies. All executable code uses standard library modules only.
- The user-facing "product" is mostly content: 19 command wrappers in `plugins/novel/commands/`, 16 workflow definitions in `plugins/novel/workflows/`, 20 skills in `plugins/novel/skills/`, 6 agent prompts in `plugins/novel/agents/`, and 10 templates in `plugins/novel/templates/`.
- The effective runtime is hybrid:
  - Node handles install-time translation and validation
  - Python handles authoring-time state management and import/promotion helpers
